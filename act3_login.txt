Vamos a desarrollar a fondo el Bloque de Autenticación y Autorización (Criterio 1 - 40%), el cual equivale a 4 puntos directos de la nota de tu Actividad 3.

Para cumplir estrictamente con el enunciado de la UNIR, implementaremos el patrón Phantom Token (Token Fantasma). Este patrón divide la seguridad en dos niveles:

Hacia el exterior (Frontend SPA en React): Se expone un Token Opaco (un string aleatorio/UUID sin significado semántico), impidiendo que se filtren los roles o datos del usuario en el navegador del cliente.

Hacia el interior (Malla de Microservicios): El API Gateway intercepta ese Token Opaco, lo valida de forma ultra veloz consultando una caché en memoria (Redis) y lo intercambia por un JWT firmado criptográficamente (El Fantasma). Este JWT se inyecta en la cabecera HTTP y viaja hacia los microservicios (book-catalogue, orders-service, etc.), permitiéndoles validar de manera local y sin estado los roles y claims.

2. Lógica del Generador de Tokens (TokenService.java):
Este componente se encarga de fabricar ambas estructuras de tokens y empaquetarlas en Redis.

Java
package com.unir.users.service;

import lombok.RequiredArgsConstructor;
import org.springframework.data.redis.core.StringRedisTemplate;
import org.springframework.security.oauth2.jwt.JwtClaimsSet;
import org.springframework.security.oauth2.jwt.JwtEncoder;
import org.springframework.security.oauth2.jwt.JwtEncoderParameters;
import org.springframework.stereotype.Service;

import java.time.Instant;
import java.time.temporal.ChronoUnit;
import java.util.List;
import java.util.UUID;
import java.util.concurrent.TimeUnit;

@Service
@RequiredArgsConstructor
public class TokenService {

    private final StringRedisTemplate redisTemplate; // Cliente de operaciones para Redis
    private final JwtEncoder jwtEncoder; // Codificador inyectado con llaves asimétricas RSA

    public String generatePhantomTokenPattern(String username, String role, Long userId) {
        Instant now = Instant.now();

        // 1. Crear el JWT Interno (Estructura pesada con metadatos y roles)
        JwtClaimsSet claims = JwtClaimsSet.builder()
                .issuer("relatos-papel-auth")
                .issuedAt(now)
                .expiresAt(now.plus(5, ChronoUnit.MINUTES)) // Tiempo corto de vida útil
                .subject(userId.toString())
                .claim("username", username)
                .claim("roles", List.of(role)) // Colección de roles, ej: "ROLE_LECTOR"
                .build();

        String internalJwt = this.jwtEncoder.encode(JwtEncoderParameters.from(claims)).getTokenValue();

        // 2. Crear el Token Opaco Exterior (Referencia ciega tipo UUID sin semántica)
        String opaqueToken = "rp_opaque_" + UUID.randomUUID().toString().replace("-", "");

        // 3. Persistir en la base de datos de Redis en RAM con TTL idéntico (300 segundos)
        // Estructura en Redis -> Key: rp_opaque_xxxx... | Value: eyJhbGciOiJSUzI1Ni... (JWT)
        redisTemplate.opsForValue().set(opaqueToken, internalJwt, 5, TimeUnit.MINUTES);

        // Retornamos únicamente el Token Opaco al controlador para que viaje al Frontend React
        return opaqueToken;
    }
}

2. Filtro de Intercambio de Tokens (PhantomTokenGatewayFilter.java):
Este componente automatiza de forma transparente la sustitución del token de cara al backend interno.

Java
package com.unir.gateway.filter;

import lombok.RequiredArgsConstructor;
import org.springframework.cloud.gateway.filter.GatewayFilterChain;
import org.springframework.cloud.gateway.filter.GlobalFilter;
import org.springframework.core.Ordered;
import org.springframework.data.redis.core.StringRedisTemplate;
import org.springframework.http.HttpHeaders;
import org.springframework.http.HttpStatus;
import org.springframework.http.server.reactive.ServerHttpRequest;
import org.springframework.http.server.reactive.ServerHttpResponse;
import org.springframework.stereotype.Component;
import org.springframework.web.server.ServerWebExchange;
import reactor.core.publisher.Mono;

@Component
@RequiredArgsConstructor
public class PhantomTokenGatewayFilter implements GlobalFilter, Ordered {

    private final StringRedisTemplate redisTemplate; // Consultas directas a Redis en RAM

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        ServerHttpRequest request = exchange.getRequest();
        String path = request.getURI().getPath();

        // 1. Excluir endpoints públicos de autenticación para evitar bucles infinitos
        if (path.startsWith("/api/v1/auth/")) {
            return chain.filter(exchange);
        }

        // 2. Extraer la cabecera normalizada de autorización HTTP
        String authHeader = request.getHeaders().getFirst(HttpHeaders.AUTHORIZATION);
        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            return onError(exchange, HttpStatus.UNAUTHORIZED); // Petición no identificada
        }

        String externalOpaqueToken = authHeader.substring(7);

        // 3. Golpe de resolución asíncrona a la caché en memoria de Redis
        String internalJwt = redisTemplate.opsForValue().get(externalOpaqueToken);

        if (internalJwt == null) {
            // Si el token expiró en Redis o no existe, se rechaza en el perímetro de forma instantánea
            return onError(exchange, HttpStatus.UNAUTHORIZED);
        }

        // 4. PASO CLAVE: Sobrescribir la cabecera mutando la petición.
        // Eliminamos el token opaco e inyectamos el JWT real en la cabecera 'Authorization' hacia los microservicios internos.
        ServerHttpRequest mutatedRequest = request.mutate()
                .header(HttpHeaders.AUTHORIZATION, "Bearer " + internalJwt)
                .build();

        return chain.filter(exchange.mutate().request(mutatedRequest).build());
    }

    private Mono<Void> Mono<Void> onError(ServerWebExchange exchange, HttpStatus status) {
        ServerHttpResponse response = exchange.getResponse();
        response.setStatusCode(status);
        return response.setComplete();
    }

    @Override
    public int getOrder() {
        return -1; // Máxima precedencia de ejecución en la pila de filtros
    }
}
🍃 C. Consumo y Validación de Grano Fino en los Microservicios (Ej: book-catalogue)
Gracias al filtro perimetral del Gateway, el microservicio de negocio no tiene idea de la existencia del Token Opaco; él recibe un JWT tradicional firmado de forma limpia. Activamos Spring Security localmente para validar los claims y autorizar por métodos.

1. Configuración de Seguridad Local (ResourceServerConfig.java):
Java
package com.unir.products.config;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.security.config.annotation.method.configuration.EnableMethodSecurity;
import org.springframework.security.config.annotation.web.builders.HttpSecurity;
import org.springframework.security.config.annotation.web.configuration.EnableWebSecurity;
import org.springframework.security.oauth2.server.resource.authentication.JwtAuthenticationConverter;
import org.springframework.security.oauth2.server.resource.authentication.JwtGrantedAuthoritiesConverter;
import org.springframework.security.web.SecurityFilterChain;

@Configuration
@EnableWebSecurity
@EnableMethodSecurity // Habilita anotaciones fuertes como @PreAuthorize
public class ResourceServerConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable()) // Stateless API
            .authorizeHttpRequests(auth -> auth
                .anyRequest().authenticated() // Toda petición que entra debe traer un JWT válido
            )
            .oauth2ResourceServer(oauth2 -> oauth2.jwt(jwt ->
                jwt.jwtAuthenticationConverter(jwtAuthenticationConverter())
            ));
        return http.build();
    }

    // Extractor de Autoridades: Transforma el claim custom "roles" del JWT al formato de Spring
    private JwtAuthenticationConverter jwtAuthenticationConverter() {
        JwtGrantedAuthoritiesConverter authoritiesConverter = new JwtGrantedAuthoritiesConverter();
        authoritiesConverter.setAuthoritiesClaimName("roles");
        authoritiesConverter.setAuthorityPrefix(""); // Evitamos duplicar prefijos redundantes

        JwtAuthenticationConverter converter = new JwtAuthenticationConverter();
        converter.setJwtGrantedAuthoritiesConverter(authoritiesConverter);
        return converter;
    }
}

"Implementamos el patrón Phantom Token para cumplir con el principio de Defensa en Profundidad. Si usáramos tokens opacos de extremo a extremo, saturaríamos el sistema con llamadas de introspección de red síncronas contra el servidor de usuarios. Al convertirlos en el Gateway en JWTs internos, permitimos que la malla de microservicios valide los permisos localmente en memoria sin golpear bases de datos, garantizando una latencia constante de O(1) en cada petición."

- No metas consultas bloqueantes a Redis dentro del filtro.

PASO 16. Qué explicar en la memoria técnica
---------------------------------------------------------------------

Puntos que conviene dejar muy claros:

1) El frontend no recibe roles ni claims sensibles.
2) El token externo es opaco y no revela información.
3) El Gateway actúa como punto de intercambio de tokens.
4) Redis evita introspecciones repetidas y reduce latencia.
5) Los microservicios validan el JWT de forma local y stateless.
6) La autorización fina se hace por roles y, si da tiempo, por recurso.

Frase útil para la memoria:

"Se implementa el patrón Phantom Token para separar la seguridad externa de la interna. El cliente opera con un token opaco sin semántica, mientras que el API Gateway lo intercambia por un JWT interno firmado, permitiendo que los microservicios validen permisos localmente sin depender de introspecciones remotas en cada petición."

=====================================================================
ATENCIÓN ESPECIAL: PROBLEMA REAL ENTRE `ms-orders` Y `ms-catalogue`
=====================================================================

Este punto es MUY importante porque puede romper tu autenticación si no lo tienes en cuenta.

He revisado que `ms-orders` llama directamente a `catalogue` usando `RestTemplate` con balanceo de carga:

- archivo: `ms-orders/src/main/java/com/unir/papertales/orders/client/CatalogueClient.java`
- URL usada: `http://catalogue/api/v1/books/{bookId}`

Esto significa que:

1) `ms-orders` NO entra al `api-gateway` para esa llamada interna.
2) Por tanto, el patrón Phantom Token del Gateway NO participa en esa comunicación.
3) Si proteges `ms-catalogue` con JWT, `ms-orders` tendrá que enviar también un JWT válido cuando invoque a `catalogue`.

Tienes tres estrategias posibles:

ESTRATEGIA A. Propagar el JWT del usuario autenticado
------------------------------------------------------

La más coherente para una práctica académica avanzada.

Idea:

- El usuario entra por Gateway con token opaco.
- El Gateway lo cambia por JWT y se lo pasa a `ms-orders`.
- `ms-orders` reutiliza ese JWT cuando llama a `catalogue`.

Ventajas:

- Mantienes trazabilidad del usuario real.
- La autorización sigue siendo extremo a extremo dentro de la malla.

Inconveniente:

- Hay que capturar la cabecera `Authorization` y reenviarla en `CatalogueClient`.

ESTRATEGIA B. Permitir comunicación de servicio a servicio con una excepción controlada
----------------------------------------------------------------------------------------

Idea:

- `ms-catalogue` acepta ciertas llamadas internas desde `ms-orders` por una ruta separada o una regla especial.

Ventajas:

- Implementación más rápida.

Inconvenientes:

- Menos elegante.
- Más difícil de justificar si te preguntan por seguridad homogénea.

ESTRATEGIA C. Hacer que `ms-orders` invoque catálogo a través del Gateway
-------------------------------------------------------------------------

Idea:

- En lugar de llamar a `http://catalogue/...`, llamas al Gateway.

Inconvenientes:

- No es lo más limpio para comunicación interna entre microservicios.
- Puede introducir acoplamiento innecesario.

RECOMENDACIÓN PARA TU ACTIVIDAD:
--------------------------------

La mejor opción para defender el diseño es la ESTRATEGIA A: propagar el JWT interno desde `ms-orders` hacia `ms-catalogue`.

Eso implica que en `CatalogueClient` tendrás que:

1) leer la cabecera Authorization de la petición entrante o del contexto de seguridad,
2) copiarla en la llamada saliente hacia `catalogue`,
3) dejar que `ms-catalogue` valide ese mismo JWT.

Si no haces esto, es muy probable que `ms-orders` deje de funcionar en las llamadas internas una vez actives seguridad en catálogo.
